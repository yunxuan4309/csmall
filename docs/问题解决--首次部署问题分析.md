## 阻塞性问题（必须修复才能上线）

### P1：4 个业务模块缺少生产配置文件

**现象**：

以下 4 个模块没有 `application-prod.yml` 文件：
- `mall-order`（订单服务）
- `mall-seckill`（秒杀服务）
- `mall-search`（搜索服务）
- `mall-front`（前台服务）

**为什么是问题**：

Spring Boot 的 profile 机制是：`application.yml`（基础配置）→ `application-{profile}.yml`（环境配置覆盖）。

当你用 `--spring.profiles.active=prod` 启动服务时，Spring Boot 会尝试加载 `application-prod.yml`。如果这个文件不存在，服务只能使用 `application.yml` 中的基础配置，而基础配置里写的是 `spring.profiles.active: test`，所以：

1. 数据库连接会指向 `localhost`
2. Nacos 会连 `localhost:8848`
3. Seata 会连 `localhost:8091`
4. 这些在云服务器上都是不可用的

**怎么解决**：

参考已有模块（如 `mall-ams`）的 `application-prod.yml`，为每个缺失的模块创建生产配置文件。以 `mall-order` 为例，需要创建：

`mall-order/mall-order-webapi/src/main/resources/application-prod.yml`

```yaml
my:
  server:
    addr: <你的阿里云服务器公网IP>

spring:
  datasource:
    url: jdbc:mysql://${my.server.addr}:3306/cs_mall_oms?useSSL=false&useUnicode=true&characterEncoding=utf-8&serverTimezone=Asia/Shanghai&allowMultiQueries=true&allowPublicKeyRetrieval=true
    username: <生产数据库用户名>
    password: <生产数据库密码>
    type: com.alibaba.druid.pool.DruidDataSource
    driver-class-name: com.mysql.cj.jdbc.Driver
  application:
    name: mall-order
  cloud:
    nacos:
      discovery:
        server-addr: ${my.server.addr}:8848
  redis:
    host: ${my.server.addr}
    port: 6379
    password: <Redis密码>

dubbo:
  application:
    name: mall-order-dubbo
  protocol:
    port: -1
    name: dubbo
  registry:
    address: nacos://${my.server.addr}:8848
    use-as-config-center: false
    use-as-metadata-center: false
  consumer:
    check: false
    timeout: 5000

seata:
  tx-service-group: csmall_group
  service:
    vgroup-mapping:
      csmall_group: default
    grouplist:
      default: ${my.server.addr}:8091
  client:
    undo:
      log-serialization: jackson

logging:
  level:
    com.cooxiao.mall: info
```

其他 3 个模块按同样模式创建，注意替换各自的数据库名和服务名。

### P2：Gateway 生产配置指向内网 IP

**现象**：

文件：`mall-gateway-server/src/main/resources/application-prod.yml`

```yaml
my:
  server:
    addr: 192.168.137.150    # ← 这是一个内网/虚拟机 IP
    #addr: 106.75.107.22     # ← 云服务器 IP 被注释了
```

**为什么是问题**：

`192.168.x.x` 是 RFC 1918 私有地址，只能在局域网内访问。部署到阿里云后：
- Gateway 需要通过这个地址连接 Nacos 进行服务发现
- 如果 Nacos 部署在同一台云服务器上，应使用 `127.0.0.1` 或云服务器的私有 IP
- 如果 Nacos 部署在其他机器上，应使用对应的内网 IP（阿里云 VPC 内网）
- `106.75.107.22` 看起来是公网 IP，服务间通信走公网会有额外延迟和费用

**怎么解决**：

根据你的部署方案选择正确地址：

- **单机部署**（所有组件在一台 ECS 上）：使用 `127.0.0.1`
- **多机部署**：使用阿里云 **VPC 内网 IP**（不是公网 IP），内网通信免费且低延迟

```yaml
my:
  server:
    addr: 127.0.0.1    # 单机部署时用本机地址
```

> 注意：阿里云 ECS 有「公网 IP」和「私有 IP」两个地址。服务间通信（如 Gateway → Nacos、Dubbo → Nacos）应走**私有 IP**，只有用户访问入口才走公网 IP。

### P3：文件上传路径是 Windows 路径，Linux 无法使用

**现象**：

文件：`mall-resource/src/main/resources/application-prod.yml`

```yaml
custom:
  file-upload:
    server-local-base-path: E:/csmall-upload    # ← Windows 绝对路径
```

**为什么是问题**：

阿里云 ECS 默认操作系统是 Linux，Linux 没有盘符概念（没有 `E:` 盘）。这个路径在 Linux 上会被解释为相对路径 `E:/csmall-upload`（当前目录下的 `E:` 文件夹），导致文件上传失败或存储位置不可预期。

**怎么解决**：

将路径改为 Linux 标准路径：

```yaml
custom:
  file-upload:
    server-local-base-path: /data/csmall-upload    # Linux 标准数据目录
```

然后在服务器上提前创建该目录并赋予权限：

```bash
sudo mkdir -p /data/csmall-upload
sudo chown -R <运行服务的用户>:<用户组> /data/csmall-upload
```

> 提示：如果你的服务器磁盘不够大，可以考虑将上传目录挂载到 OSS（阿里云对象存储），但那是后续优化，初次上线用本地磁盘即可。

### P4：文件资源 URL 指向 localhost，用户无法访问图片

**现象**：

文件：`mall-resource/src/main/resources/application-prod.yml`

```yaml
custom:
  file-upload:
    resource-host: http://localhost:9060/    # ← localhost 在用户浏览器上解析为用户自己的电脑
```

**为什么是问题**：

这个配置用于生成上传文件后的访问 URL。当用户上传商品图片后，系统会返回类似 `http://localhost:9060/picuture/xxx.jpg` 的 URL。但 `localhost` 在用户浏览器中指向的是**用户自己的电脑**，而不是你的服务器，所以图片永远加载不出来。

**怎么解决**：

将 `resource-host` 改为你的域名或公网 IP：

```yaml
custom:
  file-upload:
    # 使用域名（推荐，后续配 HTTPS 更方便）
    resource-host: http://你的域名/
    # 或者使用公网 IP
    # resource-host: http://你的阿里云公网IP:9060/
```

> 注意：如果通过 Nginx 反向代理（推荐），端口可能不需要写，因为 Nginx 监听 80 端口即可。例如 `http://你的域名/resource/` 然后在 Nginx 中配置 `/resource/` 路径代理到 `localhost:9060`。

### P5：UMS 生产配置中 profiles.active 错误回退到 dev

**现象**：

文件：`mall-ums/mall-ums-webapi/src/main/resources/application-prod.yml` 第 19-20 行

```yaml
spring:
  profiles:
    active: dev    # ← 在 prod 配置文件里写了 active: dev
```

**为什么是问题**：

Spring Boot 的 profile 加载顺序是：先加载 `application.yml`，再用 `application-{profile}.yml` 覆盖。当你在 `application-prod.yml` 中写 `spring.profiles.active: dev` 时：

1. 你用 `--spring.profiles.active=prod` 启动
2. Spring Boot 加载 `application-prod.yml`
3. 里面又声明了 `active: dev`
4. Spring Boot 会额外加载 `application-dev.yml`，用 dev 的配置覆盖 prod 的配置
5. 结果：数据库地址变成 dev 环境的地址，密码也变回 dev 的密码

这相当于 prod 环境实际上跑的是 dev 配置，完全失去了生产隔离。

**怎么解决**：

删除 `application-prod.yml` 中的 `spring.profiles.active` 行：

```yaml
spring:
  # profiles:        ← 删除这整段
  #   active: dev    ← 删除这整段
  application:
    name: mall-ums
```

profile 的激活应该在启动命令中指定，而不是在 profile 文件内部循环引用：

```bash
java -jar mall-ums-webapi.jar --spring.profiles.active=prod
```

> 排查：检查其他模块的 `application-prod.yml` 是否也有同样的问题，确保只在 `application.yml` 基础配置中写一次 `spring.profiles.active`。

## 安全性问题（上线前必须处理）

### S1：JWT Secret 是明文可猜测字符串且提交到了 Git

**现象**：

所有模块的 `application.yml` 中：

```yaml
jwt:
  secret: CooxiaoMall2026JwtSecretKeyForHs512AlgorithmMustBeAtLeast64BytesLong
```

这段密钥是一个人类可读的英文句子，而且已经提交到了 GitHub 公开仓库。

**为什么是问题**：

1. **可猜测**：这个密钥由项目名+年份+用途描述组成，不是随机生成的，攻击者可以通过字典攻击或社工方式猜到
2. **已泄露**：密钥在 Git 历史中，即使现在改了，旧 commit 中仍然可见
3. **全环境共用**：dev/test/prod 用同一个密钥，一旦泄露所有环境都不安全
4. **后果**：攻击者拿到密钥后可以伪造任意用户的 JWT Token，获得管理员权限

**怎么解决**：

**第一步**：生成一个安全的随机密钥（HS512 需要 >= 64 字节）：

```bash
# Linux/Mac
openssl rand -base64 64 | head -c 64
# 或 Java
# java -e 'import secrets; print(secrets.token_urlsafe(64))'
```

**第二步**：不要把新密钥写回 yml 文件，而是通过以下方式注入：

方案 A（推荐）：通过 Nacos 配置中心管理
- 在 Nacos 控制台中添加 `jwt.secret` 配置
- yml 文件中只写占位或默认值

方案 B：通过环境变量注入
```yaml
jwt:
  secret: ${JWT_SECRET:默认开发用密钥}
```
启动时：`JWT_SECRET=真实密钥 java -jar app.jar`

方案 C：通过启动参数覆盖
```bash
java -jar app.jar --jwt.secret=真实密钥
```

**第三步**：确保新密钥不再提交到 Git。

### S2：MySQL 密码明文写入配置文件并提交到 Git

**现象**：

多个模块的 `application-prod.yml` 中：

```yaml
spring:
  datasource:
    username: root
    password: tarena2017Up;
```

**为什么是问题**：

1. 数据库密码直接写在配置文件中，任何人 clone 仓库都能看到
2. `root` 账户拥有数据库全部权限，一旦被入侵后果最严重
3. Git 历史中永久保留，即使删除后重新提交也能从历史中找到

**怎么解决**：

**短期**（上线前必须做）：
1. 在云服务器上创建专用数据库用户，只授予必要的库权限：

```sql
-- 创建专用用户
CREATE USER 'csmall_app'@'%' IDENTIFIED BY '一个强随机密码';

-- 只授权给需要的数据库
GRANT SELECT, INSERT, UPDATE, DELETE ON cs_mall_pms.* TO 'csmall_app'@'%';
GRANT SELECT, INSERT, UPDATE, DELETE ON cs_mall_oms.* TO 'csmall_app'@'%';
GRANT SELECT, INSERT, UPDATE, DELETE ON cs_mall_ums.* TO 'csmall_app'@'%';
GRANT SELECT, INSERT, UPDATE, DELETE ON cs_mall_ams.* TO 'csmall_app'@'%';
GRANT SELECT, INSERT, UPDATE, DELETE ON cs_mall_seckill.* TO 'csmall_app'@'%';
FLUSH PRIVILEGES;
```

2. 用环境变量或 Nacos 配置中心管理密码（同 S1 的方案）

**中期**（有精力时做）：
- 考虑使用阿里云 RDS MySQL 托管数据库，自带访问控制、自动备份、SSL 连接

### S3：MySQL 使用 root 账户且 useSSL=false

**现象**：

```yaml
spring:
  datasource:
    username: root
    url: jdbc:mysql://...?useSSL=false&...
```

**为什么是问题**：

1. **root 账户**：拥有所有数据库的所有权限（包括 DROP DATABASE、GRANT 等），如果应用被 SQL 注入攻击，攻击者可以删库
2. **useSSL=false**：数据库连接不加密，在公网传输时数据（包括密码）可以被中间人截获

**怎么解决**：

1. 创建专用数据库用户（见 S2），禁用 root 远程登录：

```sql
-- 禁止 root 远程登录
DELETE FROM mysql.user WHERE User='root' AND Host NOT IN ('localhost', '127.0.0.1', '::1');
FLUSH PRIVILEGES;
```

2. 如果 MySQL 和应用在同一台机器上，`useSSL=false` 可以接受（走本机回环，不经过网络）。但如果 MySQL 是远程的，必须开启 SSL：

```yaml
url: jdbc:mysql://...?useSSL=true&requireSSL=true&...
```

或使用阿里云 RDS，默认提供 SSL。

### S4：Redis 生产环境无密码

**现象**：

```yaml
spring:
  redis:
    host: ${my.server.addr}
    port: 6379
    password:          # ← 空密码
```

**为什么是问题**：

Redis 默认无密码，如果端口暴露在公网上，任何人都可以连接你的 Redis：
- 读取所有缓存数据（包括用户 Session、Token 黑名单）
- 写入恶意数据
- 执行 `FLUSHALL` 清空所有数据
- 利用 Redis 写 SSH 公钥实现服务器入侵（这是真实发生过的大规模攻击方式）

**怎么解决**：

1. 在 Redis 配置文件中设置密码：

```bash
# 编辑 redis.conf
requirepass 你的强随机密码
```

2. 在 yml 中配置密码（建议通过环境变量或 Nacos 注入）：

```yaml
spring:
  redis:
    host: ${my.server.addr}
    port: 6379
    password: ${REDIS_PASSWORD:}
```

3. **关键**：在阿里云安全组中，**不要**把 6379 端口开放给公网，只允许本机（127.0.0.1）或 VPC 内网访问。

### S5：CORS 配置为 `*`，允许任意来源跨域

**现象**：

文件：`mall-sso/src/main/java/com/cooxiao/mall/sso/security/config/SSOWebSecurityConfig.java`

```java
configuration.setAllowedOriginPatterns(Arrays.asList("*"));   // 允许任何来源
configuration.setAllowCredentials(true);                       // 同时允许携带凭证
```

其他服务（AMS、UMS、Product 等）也有相同的 CORS 配置。

**为什么是问题**：

CORS（跨域资源共享）控制的是「哪些网站的前端 JS 可以调用你的 API」。设置为 `*` 意味着：
- 任何恶意网站都可以通过用户浏览器调用你的 API
- 配合 `allowCredentials(true)`，恶意网站可以携带用户的 Cookie/Token
- 这为 CSRF（跨站请求伪造）攻击打开了大门

**怎么解决**：

将 `*` 替换为你的实际域名：

```java
configuration.setAllowedOriginPatterns(Arrays.asList(
    "https://你的域名",
    "http://你的域名",      // 开发阶段
    "http://localhost:*"     // 仅本地开发
));
```

> 注意：`allowedOrigins("*")` 和 `allowCredentials(true)` 的组合在 Spring Framework 6+ 中已被禁止，但 `allowedOriginPatterns("*")` 仍允许。这是 Spring 的一个设计妥协，不代表它是安全的。

### S6：Gateway 动态路由暴露所有服务，包括后台管理

**现象**：

文件：`mall-gateway-server/src/main/resources/application-prod.yml`

```yaml
spring:
  cloud:
    gateway:
      discovery:
        locator:
          enabled: true    # ← 所有注册到 Nacos 的服务都自动创建路由
```

**为什么是问题**：

`locator.enabled: true` 会为 Nacos 中注册的每个服务自动创建路由规则，路径格式为 `/<service-name>/**`。这意味着：

- `/mall-ams/**` → 后台管理系统对外暴露
- `/mall-ums/**` → 用户管理服务对外暴露
- `/mall-seckill/**` → 秒杀内部服务对外暴露
- 等等...

所有服务都暴露在公网上，攻击者可以直接访问后台管理接口。

**怎么解决**：

关闭动态路由，改为手动定义只允许外部访问的路由：

```yaml
spring:
  cloud:
    gateway:
      discovery:
        locator:
          enabled: false    # 关闭动态路由
      routes:
        # 只暴露前台用户需要访问的服务
        - id: mall-front
          uri: lb://mall-front
          predicates:
            - Path=/front/**
        - id: mall-order
          uri: lb://mall-order
          predicates:
            - Path=/oms/**
        - id: mall-search
          uri: lb://mall-search
          predicates:
            - Path=/search/**
        - id: mall-sso
          uri: lb://mall-sso
          predicates:
            - Path=/sso/**
        - id: mall-resource
          uri: lb://mall-resource
          predicates:
            - Path=/resource/**
        # mall-ams（后台管理）不添加路由，不对外暴露
        # mall-ums, mall-product 等内部服务通过 Dubbo RPC 调用，不需要 HTTP 路由
```

### S7：SSO 会话管理策略不符合 JWT 无状态设计

**现象**：

文件：`mall-sso/src/main/java/com/cooxiao/mall/sso/security/config/SSOWebSecurityConfig.java`

```java
http.sessionManagement(session -> {
    session.sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED);  // ← 不是 STATELESS
    session.sessionFixation().none();                                   // ← 不防御会话固定攻击
});
```

**为什么是问题**：

你的项目使用 JWT Token 进行认证，JWT 的设计理念是**无状态**——服务端不保存会话，所有认证信息都在 Token 中。但 `IF_REQUIRED` 允许 Spring Security 创建 HttpSession，这会导致：

1. SSO 服务可能同时依赖 Session 和 JWT，行为不可预测
2. `sessionFixation().none()` 不做任何会话固定攻击防护，登录后 Session ID 不变，攻击者可劫持会话
3. 在多实例部署时，Session 默认保存在内存中，不同实例间不共享，导致用户有时登录有时未登录

**怎么解决**：

```java
http.sessionManagement(session -> {
    session.sessionCreationPolicy(SessionCreationPolicy.STATELESS);  // 完全无状态
});
```

因为 JWT 本身就是无状态的，不需要 HttpSession。Token 的失效通过 Redis 黑名单实现即可。

### S8：各模块 Knife4j 生产环境未屏蔽

**现象**：

以 `mall-order` 的 `application.yml` 为例：

```yaml
knife4j:
  enable: true
  production: false    # ← 未屏蔽，生产环境仍然可以访问 API 文档
```

目前只有 `mall-resource` 的 `application-prod.yml` 设置了 `knife4j.production: true`。

**为什么是问题**：

Knife4j/API 文档在生产环境暴露会：
- 泄露所有 API 接口的参数、返回值、调用方式
- 攻击者可以据此构造精准攻击请求
- 可能暴露内部业务逻辑和数据结构

**怎么解决**：

在每个模块的 `application-prod.yml` 中添加：

```yaml
knife4j:
  production: true    # 生产环境屏蔽 API 文档
```

## 基础设施问题（建议完善）

### I1：无 Docker/容器化方案

**现状**：项目没有 Dockerfile 和 docker-compose.yml。

**为什么建议做**：

- 手动部署 11 个 Java 服务 + 7 个中间件，启动顺序、端口冲突、依赖关系全靠记忆和脚本管理，容易出错
- Docker Compose 可以一键启动所有组件，声明依赖关系
- 后续迁移、扩容、回滚都更方便

**建议方案**：

初次上线可以不用 Docker，直接用 systemd 管理 Java 进程。后续再逐步容器化。关键步骤：

1. 每个模块 `mvn package` 打出 jar 包
2. 上传到服务器
3. 用 systemd service 文件管理启停

```ini
# /etc/systemd/system/mall-gateway.service 示例
[Unit]
Description=CoolShark Gateway Server
After=network.target

[Service]
Type=simple
User=csmall
WorkingDirectory=/opt/csmall
ExecStart=/usr/bin/java -jar /opt/csmall/mall-gateway-server.jar --spring.profiles.active=prod
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
```

---

### I2：无 CI/CD 自动化构建部署

**现状**：没有 GitHub Actions、Jenkins 或任何自动化构建流水线。

**为什么建议做**：

手动 `mvn package` + `scp` 上传 + 重启服务，每次部署耗时且容易遗漏步骤。

**建议方案**：

初次上线可以手动部署。后续考虑 GitHub Actions：

```yaml
# .github/workflows/deploy.yml 简易示例
name: Deploy
on:
  push:
    branches: [master]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
      - run: mvn package -DskipTests
      - run: scp target/*.jar user@server:/opt/csmall/
      - run: ssh user@server 'sudo systemctl restart mall-gateway mall-sso ...'
```

### I3：无健康检查端点（Actuator 未启用）

**现状**：父 POM 中声明了 `spring-boot-starter-actuator` 依赖管理，但没有模块实际引入。

**为什么建议做**：

- 没有健康检查，你无法知道服务是否正常运行
- 阿里云 SLB（负载均衡）需要健康检查端点来判断服务状态
- Kubernetes 也依赖健康检查端点

**建议方案**：

在每个 webapi 模块中引入 actuator 依赖：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

在 `application-prod.yml` 中配置：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info    # 只暴露 health 和 info
  endpoint:
    health:
      show-details: never       # 不暴露内部细节
```

---

### I4：无 HTTPS/SSL 配置

**现状**：所有服务都是 HTTP 明文传输。

**为什么是问题**：

- HTTP 明文传输，用户密码、Token 等敏感信息可被中间人截获
- 微信支付、支付宝等第三方接入要求 HTTPS
- Chrome 浏览器对 HTTP 网站标记「不安全」，影响用户信任
- 中国法律要求网站启用 HTTPS（等保合规）

**建议方案**：

在 Nginx 层配置 HTTPS（不需要每个 Java 服务单独配）：

1. 购买域名后在阿里云申请免费 SSL 证书（Let's Encrypt 或阿里云免费证书）
2. Nginx 监听 443 端口，SSL 终止
3. Nginx 反向代理到 Gateway（10087 端口）

```nginx
server {
    listen 443 ssl;
    server_name 你的域名;

    ssl_certificate     /etc/nginx/ssl/你的域名.pem;
    ssl_certificate_key /etc/nginx/ssl/你的域名.key;

    location / {
        proxy_pass http://127.0.0.1:10087;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }
}

# HTTP 自动跳转 HTTPS
server {
    listen 80;
    server_name 你的域名;
    return 301 https://$host$request_uri;
}
```

### I5：无集中日志和监控方案

**现状**：各服务独立输出日志到本地文件，没有集中采集。

**为什么建议做**：

11 个服务分散在多个 jar 包中，排查问题时需要逐个登录服务器查看日志，效率极低。

**建议方案**：

初次上线可以暂时不做，用简单的日志文件管理：

```yaml
# application-prod.yml 中配置日志滚动
logging:
  file:
    name: /var/log/csmall/${spring.application.name}.log
  logback:
    rollingpolicy:
      max-file-size: 100MB
      max-history: 7
```

后续考虑 ELK（Elasticsearch + Logstash + Kibana）或阿里云 SLS（日志服务）。

## 上线步骤建议（阿里云）

### 第一步：购买资源

| 资源 | 规格 | 预估月费 | 说明 |
|------|------|----------|------|
| ECS 云服务器 | 2核 8GB | ¥150-200 | 新用户有折扣，可先年付 |

| MySQL | 自建 | ¥0 | 初次上线自建即可，后续迁移 RDS |

### 第二步：服务器初始化

1. 安装 JDK 21、MySQL 8、Redis、Nacos、Seata、Elasticsearch、RabbitMQ
2. 配置阿里云安全组：只开放 22（SSH）、80（HTTP）、443（HTTPS）
3. 初始化数据库，导入 `database/` 目录下的 SQL 脚本
4. 创建专用数据库用户并授权

### 第三步：修复项目问题

按本文档 P1-P5 → S1-S8 的顺序修复所有问题。

### 第四步：构建和部署

1. 本地 `mvn clean package -DskipTests`
2. 上传所有 jar 包到服务器 `/opt/csmall/`
3. 用 systemd 逐个启动服务（注意启动顺序：Nacos → MySQL → Redis → 其他中间件 → 业务服务 → Gateway）

### 第五步：域名和 HTTPS

2. 备案通过后域名解析到 ECS 公网 IP

### 第六步：验证

- 访问 `http://你的域名` 确认首页可访问

- 检查图片上传和显示是否正常
- 确认 API 文档在生产环境不可访问

## 阿里云安全组建议配置

| 协议 | 端口 | 来源 | 用途 |
|------|------|------|------|
| TCP | 22 | 你的办公 IP | SSH 远程管理 |
| TCP | 80 | 0.0.0.0/0 | HTTP（Nginx） |
| TCP | 443 | 0.0.0.0/0 | HTTPS（Nginx） |
| TCP | 3306 | 127.0.0.1 | MySQL（仅本机） |
| TCP | 6379 | 127.0.0.1 | Redis（仅本机） |
| TCP | 8848 | 127.0.0.1 | Nacos（仅本机） |
| TCP | 5672 | 127.0.0.1 | RabbitMQ（仅本机） |
| TCP | 9200 | 127.0.0.1 | Elasticsearch（仅本机） |
| TCP | 8091 | 127.0.0.1 | Seata（仅本机） |
| TCP | 10003-10009 | 127.0.0.1 | 业务服务（仅本机，通过 Gateway 访问） |

> **原则**：只开放用户必须访问的端口（80/443），其他所有端口只允许本机回环访问。业务服务通过 Gateway 统一对外，不需要直接暴露端口。
