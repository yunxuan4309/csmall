## 一、项目概述

### 1.1 项目背景

CoolShark（酷鲨）是一个基于 Spring Cloud Alibaba 的微服务电商平台，面向 C 端用户提供商品浏览、搜索、购物车、下单、支付、秒杀等完整购物流程，同时提供后台管理系统进行商品、订单、用户管理。此外集成了 AI 智能导购模块，支持自然语言商品问答、多商品对比和多轮对话导购。

**当前状态**: 项目已从生产环境（阿里云 ECS）回退到本地开发环境，以下内容以本地开发和测试环境为主。

1. **基础设施**: Nacos → MySQL → Redis → Seata → RabbitMQ → Elasticsearch → Sentinel

3. **Dubbo 提供者**: mall-product（核心，必须在消费者之前启动）

5. **依赖消费者**（等 order/product 就绪）: mall-seckill

## 六、关键设计决策

### 6.1 架构决策

1. **RPC 选型**: 使用 Apache Dubbo 3.3.2 替代 OpenFeign，性能更优，支持多种序列化协议
2. **分布式事务**: Seata AT 模式，`@GlobalTransactional` 注解标注核心业务流程
3. **ID 生成**: MyBatis-Plus `IdWorker`（雪花算法），自定义 XML INSERT 需手动调用 `IdWorker.getId()`（已弃用 mall-leaf）
4. **认证方式**: SSO 统一登录 + JWT Token + Spring Security 6.x，无状态设计
5. **API 文档**: Knife4j (OpenAPI 3)，生产环境 `knife4j.production: true` 屏蔽
6. **Long 精度**: 全局 Jackson 配置 `Long → String` 序列化，避免 JS 精度丢失（雪花 ID 为 19 位）
7. **Seata Dubbo Filter**: 自定义实现替代官方 bug 版本（`dubbo-filter-seata:1.0.2` 存在 branchType NPE）
8. **连接池**: 使用 HikariCP（Spring Boot 默认），**不使用** Druid（曾因 Druid 连接池导致上传请求线程挂死）

### 6.2 秒杀架构

- **库存预热**: Redis 预扣库存，减少数据库压力
- **异步落库**: RabbitMQ 消息队列异步创建订单，实现削峰
- **防缓存穿透**: 当前本地开发用 Redis Set 替代布隆过滤器（Windows Redis 不支持 RedisBloom），生产环境需切换
- **防超卖**: SQL 扣减库存加 `AND seckill_stock >= #{quantity}` 条件
- **消息积压保护**: RabbitMQ JSON 序列化替代 Java 序列化，`basicNack` 失败后抛出 RuntimeException 触发事务回滚

### 6.3 图片 URL 策略

- **存储**: 数据库只存纯相对路径（如 `spu_1_1.jpg`），不存完整 URL
- **拼接**: 服务层 `ImageUrlPrefixHelper`（`mall-product-webapi/.../utils/`）根据环境 `resource-host` 配置动态拼完整 URL
- **静态服务**: 生产环境 Nginx 直接从 `/data/csmall-upload/` 提供图片（`~* \.(jpg|png|gif|webp)$`），不经过 Java 服务栈
- **已上传图片兼容**: 工具类自动跳过已含 `http://` 前缀的路径

### 6.4 AI 智能导购架构

**搜索流水线**: 意图提取 → AI 查询扩展 → 多路召回 → 融合排序 → 流式生成

| 阶段 | 实现 | 说明 |
|------|------|------|
| 意图提取 | DeepSeek V4 Flash 解析用户意图 → 结构化 JSON | 替代正则猜预算，根治"降低2000左右"误判 |
| 查询扩展 | DeepSeek 实时扩展同义词 | "衣服" → "男装 女装 服装 鞋类" |
| 多路召回 | 关键词(IK `multi_match`) + 全文兜底 | 10 条/路，合并去重 |
| 向量检索 | 硅基流动 BGE-M3 (1024维) + ES `cosineSimilarity` | 免费，配置开关 `embedding-enabled` |
| 流式输出 | SSE (`/ai/chat/stream`) + 前端逐字渲染 | 用户实时看到 AI "打字" |
| 思考可视化 | 前端 thinking 卡片动画 | 展示流水线每个阶段 |

**核心依赖**:
- **对话模型**: DeepSeek V4 Flash（chat-completion API）
- **Embedding**: 硅基流动 BAAI/bge-m3（免费，OpenAI 兼容）
- **会话管理**: Redis 存储多轮对话上下文（TTL 过期）
- **成本控制**: `TokenBudgetService` — Redis 日计数器，全局 2 元/天预算上限
- **商品同步**: mall-product 增改商品后通过 Dubbo `ISpuSyncService` 自动通知 mall-ai 更新 ES 索引

### 6.5 环境配置管理

- Spring Boot Profile 机制：`application.yml`（基础配置）+ `application-test.yml`（本地开发）+ `application-prod.yml`（生产）
- 生产敏感信息通过环境变量注入：`MYSQL_USERNAME/PASSWORD`、`REDIS_PASSWORD`、`RABBITMQ_USERNAME/PASSWORD`、`AI_API_KEY` 等
- 各模块独立配置文件，互不影响

### 7.2 前后端接口对接方式

- **开发环境**: Vite 代理将前端请求按路径分发到后端 Gateway (10087)，Gateway 再路由到各微服务
- **请求封装**: `src/api/request.js` 使用 6 个独立的 Axios 实例，分别对应 SSO/Gateway/UMS/Front/Seckill/Admin 服务
- **Base URL**: 从 `.env.development` 读取 `VITE_API_*` 环境变量
- **Token 机制**: 登录后 JWT 存入 `localStorage.getItem('mall_token')`，请求拦截器自动添加 `Bearer` 前缀
- **响应格式**: 后端统一返回 `{ state: 20000, message: "ok", data: {...} }`（注意：不同服务可能使用 `20000` 或 `200`）
- **History 模式**: 路由使用 `createWebHistory`，需 Nginx 配置 `try_files $uri $uri/ /index.html` 回退

1. Element Plus 2.13.6 中 `label` 已弃用，改用 `value`
2. 空字符串可能触发后端校验，可选字段应传 `null`
3. 分页字段：不同服务可能使用 `list` 或 `records`，注意兼容
4. `request.js` 中 6 个 Axios 实例独立配置：生产环境需统一为单一 `VITE_API_BASE_URL`（或配置 Nginx 路径分发）
5. Vue 文件统一 `<script setup>` + Composition API
6. `@` 别名指向 `src` 目录

## 八、重点踩坑记录与问题解决方案

> **以下内容从多份历史文档中提取整合，是项目最宝贵的经验沉淀，不可丢弃！**

### 8.1 Spring Boot 2.x → 3.x 技术栈迁移

#### javax → jakarta 包名变更

Spring Boot 3.x 基于 Jakarta EE 9+，所有 `javax.*` 包名改为 `jakarta.*`。涉及 8 个模块的 SSOFilter、Security 配置、Controller 等 25+ 文件:

- `javax.servlet.*` → `jakarta.servlet.*`
- `javax.annotation.PostConstruct` → `jakarta.annotation.PostConstruct`
- `javax.validation.*` → `jakarta.validation.*`

#### Spring Security 6.x 配置重构

`WebSecurityConfigurerAdapter` 已移除，改为 `SecurityFilterChain` Bean + Lambda DSL：

```java
// 旧（5.x）
http.csrf().disable();
http.authorizeRequests().antMatchers("/public/**").permitAll();

// 新（6.x）
http.csrf(csrf -> csrf.disable());
http.authorizeHttpRequests(auth -> auth.requestMatchers("/public/**").permitAll());
```

关键变更：
- `@EnableGlobalMethodSecurity` → `@EnableMethodSecurity`
- `setAllowedOrigins("*")` → `setAllowedOriginPatterns("*")`
- `http.sessionManagement().sessionCreationPolicy(...)` → `http.sessionManagement(session -> session.sessionCreationPolicy(...))`

#### Gateway + Knife4j 循环依赖

`Knife4jSwaggerProvider` 使用 `@Autowired` 直接注入 `RouteLocator`，形成循环依赖。解决方案：使用 `ObjectProvider<RouteLocator>` 构造器注入 + `getIfAvailable()` 延迟加载。

### 8.2 Seata 1.x → Apache Seata 2.1.0 迁移

- **groupId 变更**: `io.seata` → `org.apache.seata`
- **关键 Bug 修复**: `dubbo-filter-seata:1.0.2` 存在 `branchType is null` NPE 问题（GitHub Issue #6815）。在消费者端无条件调用 `RootContext.getBranchType()`，当不在事务中时返回 null 导致崩溃。

**自定义 Filter 解决方案**（在 mall-common 中实现）:
```java
// SeataTransactionConsumerFilter: 先检查 xid，再安全获取 branchType
String xid = RootContext.getXID();
if (xid != null) {
    invocation.setAttachment("TX_XID", xid);
    BranchType branchType = RootContext.getBranchType();
    if (branchType != null) {
        invocation.setAttachment("TX_BRANCH_TYPE", branchType.name());
    }
}
```
通过 Dubbo SPI 注册（`META-INF/dubbo/org.apache.dubbo.rpc.Filter`）。

### 8.3 JJWT 版本升级（0.9.x → 0.12.6）

**原因**: JDK 11+ 移除 `javax.xml.bind`，JJWT 0.9.x 不兼容。

**Breaking Changes**:
| 0.9.x | 0.12.6 |
|--------|--------|
| `setClaims()` | `.claims()` |
| `signWith(key, algo)` | `.signWith(key, Jwts.SIG.HS512)` |
| `Jwts.parser().setSigningKey().parseClaimsJws().getBody()` | `Jwts.parser().verifyWith(key).build().parseSignedClaims(token).getPayload()` |
| 单个 `jjwt` 依赖 | 三个：`jjwt-api` + `jjwt-impl`(runtime) + `jjwt-jackson`(runtime) |

**密钥长度问题**: JJWT 0.12.x 严格校验 HMAC 密钥长度，HS512 要求 ≥ 64 字节。全部 9 个模块的 `jwt.secret` 替换为 69 字节新密钥。

**Token 前导空格问题**: `authHeader.substring(jwtTokenHead.length())` 截取 `"Bearer"` 后得到 `" eyJ..."`（前导空格），JJWT 0.12.x 严格校验导致解析失败。在所有 `substring(tokenHead.length())` 后加 `.trim()`，共 12 处。

### 8.4 商品模块实体类与数据库字段映射问题

**实体类缺少 @TableName**: MyBatis-Plus 默认用类名转蛇形作为表名，如 `Spu` → `spu`，但实际表名是 `pms_spu`。为 25 个实体类添加 `@TableName` 注解。

**is_ 前缀字段不匹配**: MyBatis-Plus 将 `deleted` 映射为列 `deleted`，但数据库实际列名是 `is_deleted`。为 Spu 中 5 个字段、Category 中 2 个字段添加 `@TableField` 注解。

**pms_category 表结构不一致**: Mapper XML 期望的字段（`depth`、`keywords`、`enable`）与数据库实际字段（`level`、`is_parent`、`is_display`）不匹配。

**SPU 测试数据不可见**: `init-test-data.sql` 中 SPU INSERT 未显式设置 `is_checked` 和 `is_deleted`，默认 0 导致前端查询不到商品。解决：显式设置 `is_checked=1, is_deleted=0`，并将数据从 5 条扩充到 20 条。

**未修复的遗留差异**（当前使用自定义 Mapper XML，暂不影响）：`Brand`、`Attribute`、`AttributeTemplate`、`Picture`、`SkuSpecification`、`SpuDetail` 等实体类与数据库字段存在差异，若未来使用 `BaseMapper` 方法操作需修复。

### 8.5 MyBatis-Plus 版本兼容问题

- **Starter 选择**: Spring Boot 3.x 必须用 `mybatis-plus-spring-boot3-starter`，不能用 `mybatis-plus-boot-starter`
- **分页插件 jsqlparser**: 需额外添加 `mybatis-plus-jsqlparser` 依赖
- **分页类型转换**: `IPage<Model>` 无法直接转 `IPage<VO>`，需手动 stream + `convertToVO` + new `Page<>`
- **API 变更**: `mapper.update(entity)` → `mapper.updateById(entity)`；`selectCount()` 返回 `long` 不是 `int`；代码生成器 `AutoGenerator` → `FastAutoGenerator`
- **自定义 XML 插入**: 不会经过 MyBatis-Plus `ASSIGN_ID` 逻辑，需手动 `IdWorker.getId()`

### 8.6 秒杀服务问题

**布隆过滤器不可用**: 本地 Redis 3.2 (Windows) 不支持 RedisBloom 模块 → 用 Redis Set 替代（`SADD`/`SISMEMBER`），代码留 TODO 标记，生产环境需切换。

**秒杀订单创建缓慢**: 三层问题叠加 — success 表字段不匹配 → 插入失败 → RabbitMQ `basicNack(requeue=true)` 无限重试 → `logging.level: trace` 日志刷屏 → 修复表结构 + 异常抛 RuntimeException + 日志级别改为 info。

**库存超卖**: SQL `WHERE sku_id=#{skuId}` → `WHERE sku_id=#{skuId} AND seckill_stock >= #{quantity}`，消费者检查返回值。

**success 表 id 不能为空**: 自定义 XML INSERT 不经过 MyBatis-Plus ASSIGN_ID，需手动 `IdWorker.getId()`。

**其他修复**: Redis 缓存 TTL 从 5 分钟改 2 小时 5 分钟；RabbitMQ 消息从 Java 序列化改 JSON 序列化；添加 Seata `@GlobalTransactional` 和 Sentinel 流控规则。

### 8.7 RabbitMQ Windows 环境问题

**根本原因**: Windows 计算机名含中文字符导致 Erlang 节点名乱码。`rabbitmqctl` 无法连接、AMQP 5672 不监听、管理界面 15672 不可访问。

**解决步骤**:
1. 修改计算机名为纯英文 → 重启
2. 彻底卸载 RabbitMQ + Erlang，清理残留数据（`AppData\Roaming\RabbitMQ`、`.erlang.cookie`）
3. 设置环境变量 `RABBITMQ_NODENAME=rabbit@localhost`
4. 重新安装 → 启用管理插件 → 验证

**经验**: `rabbit@nohost` 是 Erlang 无法解析主机名的典型标志；`RABBITMQ_NODENAME=rabbit@localhost` 是最稳妥方案。

### 8.8 mall-resource 模块问题

**503 路由问题**: mall-resource 无 Nacos 发现依赖，未注册到 Nacos，Gateway 的 `lb://mall-resource` 路由找不到目标。解决：改为 `http://127.0.0.1:9060` 直连。

**编译错误**: 添加上传记录功能时需引入 MySQL/MyBatis-Plus/HikariCP 依赖，注意 `mybatis-plus-jsqlparser` 不可遗漏。

**Druid 连接池 BUG**: 原使用 Druid 连接池，上传请求时可能线程挂死。解决：改为 HikariCP（Spring Boot 默认）。

**Nginx 上传代理**: 需添加 `/upload/` location 块并设置 `client_max_body_size 10m`，注意 Nginx 语法不能用嵌套 location。

**SQL 密码特殊字符**: Bash 中执行含 `!` 的密码时需用单引号包裹，防止 Bash 历史扩展。

### 8.9 Maven 多模块构建问题

- **库模块跳过 repackage**: mall-common、mall-pojo、各 service 模块需配置 `spring-boot-maven-plugin` 的 `<skip>true</skip>`，避免找不到 main class
- **子模块版本管理**: 统一通过父 POM `dependencyManagement` 管理版本，子模块引用不指定 `<version>`
- **groupId 统一**: 所有模块统一为 `com.cooxiao.mall`
- **相对路径**: 所有子模块添加 `<relativePath>../pom.xml</relativePath>`

### 8.10 Gateway 路由配置差异

| 环境 | 路由方式 | 配置 |
|------|---------|------|
| 开发/测试 (test) | 手动路由 | `locator.enabled: false`，9 条完整路由（与生产对齐） |
| 生产 (prod) | 手动路由 | `locator.enabled: false`，仅暴露前台必需服务，后台管理服务不对外 |

生产环境通过 `lb://`（需要服务在 Nacos 注册）或 `http://ip:port` 直连指定路由。

### 8.11 Git 中文乱码修复

Windows 默认 GBK 编码导致中文文件名和提交日志乱码，必须配置：
```bash
git config --global core.quotepath false
git config --global i18n.commitEncoding utf-8
git config --global i18n.logOutputEncoding utf-8
git config --global gui.encoding utf-8
```
IDEA 还需：Settings → File Encodings 全部 UTF-8，VM Options 添加 `-Dfile.encoding=UTF-8`。

### 8.12 其他经验教训

- **Druid 连接池**: 线上曾因 Druid 导致线程挂死，已全面替换为 HikariCP
- **Apache Commons Lang**: `StringUtils` 包名从 `org.apache.commons.lang` 改为 `org.apache.commons.lang3`
- **Nacos RandomUtils**: 旧版 `com.alibaba.nacos.client.naming.utils.RandomUtils` 不可用，改为 `ThreadLocalRandom`
- **MediaType.APPLICATION_JSON_UTF8**: 已废弃，RFC 8259 规定 JSON 默认 UTF-8
- **CORS 配置**: `setAllowedOrigins("*")` → `setAllowedOriginPatterns("*")` + `setAllowCredentials(true)`
- **JacksonConfig**: 全局 Long→String 序列化 + 空值处理
- **Elasticsearch**: 低配服务器需配置 JVM 堆内存（`-Xms512m -Xmx512m`），避免被 OOM Killer 杀死
