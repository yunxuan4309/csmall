# Spring Boot 2.x → 3.x 技术栈迁移问题总结

## 问题背景

本次项目从 Spring Boot 2.x 版本迁移至 Spring Boot 3.5.7 版本，涉及到 Spring Security 6.x、Jakarta EE 9+、MyBatis Plus 4.x 等组件的升级。在迁移过程中遇到了多个兼容性问题，现已全部解决。

---

## 一、Spring Security 6.x 兼容性问题

### 1.1 Servlet API 包名变更 (javax → jakarta)

**问题描述**：
```
Cannot resolve symbol 'ServletException'
Cannot resolve symbol 'HttpServletRequest'
Cannot resolve symbol 'HttpServletResponse'
```

**根本原因**：
Spring Boot 3.x 基于 **Jakarta EE 9+**，Oracle 将 Java EE 捐赠给 Eclipse 基金会后，Eclipse 基金会因商标问题将所有 `javax.*` 包名重命名为 `jakarta.*`。

**影响范围**：
- 所有使用 Servlet API 的类
- 包括 Filter、Handler、Controller 等

**解决方案**：
```java
// 修复前
import javax.servlet.FilterChain;
import javax.servlet.ServletException;
import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;

// 修复后
import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
```

**修改文件列表**：
- `mall-ams`: SSOFilter, MyAccessDeniedHandler, MyAuthenticationEntryPoint, ResourceWebSecurityConfiguration
- `mall-front`: SSOFilter, MyAccessDeniedHandler, MyAuthenticationEntryPoint, ResourceWebSecurityConfiguration
- `mall-order`: SSOFilter, MyAccessDeniedHandler, MyAuthenticationEntryPoint, ResourceWebSecurityConfiguration
- `mall-ums`: SSOFilter, MyAccessDeniedHandler, MyAuthenticationEntryPoint, ResourceWebSecurityConfiguration, LoginUtils, UserController
- `mall-product`: SSOFilter, MyAccessDeniedHandler, MyAuthenticationEntryPoint, ResourceWebSecurityConfiguration
- `mall-search`: SSOFilter, MyAccessDeniedHandler, MyAuthenticationEntryPoint, ResourceWebSecurityConfiguration
- `mall-seckill`: SSOFilter, MyAccessDeniedHandler, MyAuthenticationEntryPoint, ResourceWebSecurityConfiguration
- `mall-sso`: SSOFilter, MyAccessDeniedHandler, MyAuthenticationEntryPoint, SSOWebSecurityConfig, LoginUtils, AdminSSOController, UserSSOController

### 1.2 WebSecurityConfigurerAdapter 已移除

**问题描述**：
```
Cannot resolve class 'WebSecurityConfigurerAdapter'
```

**根本原因**：
Spring Security 5.7 开始弃用 `WebSecurityConfigurerAdapter`，6.0 完全移除了该类，推荐使用组件式配置。

**解决方案**：
```java
// 修复前
@Configuration
@EnableWebSecurity
public class ResourceWebSecurityConfiguration extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        // ...
    }
}

// 修复后
@Configuration
@EnableWebSecurity
public class ResourceWebSecurityConfiguration {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        // ...
        return http.build();
    }
}
```

### 1.3 Spring Security 配置 API 变更 (Lambda DSL)

**根本原因**：
Spring Security 6.x 强制使用 Lambda 风格的配置 API。

**API 变更对照表**：

| 旧 API (Spring Security 5.x) | 新 API (Spring Security 6.x) |
|------------------------------|------------------------------|
| `http.csrf().disable()` | `http.csrf(csrf -> csrf.disable())` |
| `http.cors().configurationSource(...)` | `http.cors(cors -> cors.configurationSource(...))` |
| `http.authorizeRequests()` | `http.authorizeHttpRequests()` |
| `http.authorizeRequests().antMatchers(...)` | `http.authorizeHttpRequests().requestMatchers(...)` |
| `http.sessionManagement().sessionCreationPolicy(...)` | `http.sessionManagement(session -> session.sessionCreationPolicy(...))` |

**示例**：
```java
// 修复前
@Override
protected void configure(HttpSecurity http) throws Exception {
    http.csrf().disable();
    http.cors().configurationSource(corsConfigurationSource());
    http.authorizeRequests()
        .antMatchers("/**").permitAll()
        .anyRequest().authenticated();
    http.sessionManagement().sessionCreationPolicy(SessionCreationPolicy.STATELESS);
    http.addFilterBefore(ssoFilter, UsernamePasswordAuthenticationFilter.class);
}

// 修复后
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.csrf(csrf -> csrf.disable());
    http.cors(cors -> cors.configurationSource(corsConfigurationSource()));
    http.authorizeHttpRequests(auth -> auth
        .requestMatchers("/**").permitAll()
        .anyRequest().authenticated()
    );
    http.sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS));
    http.addFilterBefore(ssoFilter, UsernamePasswordAuthenticationFilter.class);
    return http.build();
}
```

### 1.4 @EnableGlobalMethodSecurity 已弃用

**问题描述**：
```
Annotation @EnableGlobalMethodSecurity is marked as @Deprecated
```

**根本原因**：
Spring Security 6.x 使用 `@EnableMethodSecurity` 替代 `@EnableGlobalMethodSecurity`。

**解决方案**：
```java
// 修复前
@EnableGlobalMethodSecurity(prePostEnabled = true)

// 修复后
@EnableMethodSecurity(prePostEnabled = true)
```

## 二、MyBatis Plus 兼容性问题

### 2.1 缺少 mybatis-plus-boot-starter 依赖

**问题描述**：
```
Cannot resolve symbol 'BaseMapper'
Cannot resolve symbol 'MybatisPlusException'
```

**根本原因**：
`mall-ams-webapi` 模块的 pom.xml 中缺少 `mybatis-plus-boot-starter` 依赖，导致无法使用 `BaseMapper` 等类。

**解决方案**：
在 `mall-ams-webapi/pom.xml` 中添加依赖：
```xml
<!-- MyBatis Plus Spring Boot：包含 BaseMapper -->
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-boot-starter</artifactId>
</dependency>
<!-- mall-pojo 直接依赖 -->
<dependency>
    <groupId>com.cooxiao.mall</groupId>
    <artifactId>mall-pojo</artifactId>
</dependency>
```

同时在 `mall-pojo/pom.xml` 中添加注解依赖：
```xml
<!-- MyBatis Plus：提供实体类注解（仅编译时使用） -->
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-annotations</artifactId>
    <version>3.5.7</version>
    <scope>provided</scope>
</dependency>
```

### 2.2 mall-pojo Model 类缺少 MyBatis Plus 注解

**问题描述**：
Mapper 接口继承 `BaseMapper<T>` 但 Model 类没有 `@TableName` 等注解。

**根本原因**：
mall-pojo 中的实体类没有添加 MyBatis Plus 的注解，导致无法正确映射数据库表。

**解决方案**：
为以下 Model 类添加注解：
```java
// Admin.java
@Data
@TableName("ams_admin")
public class Admin implements Serializable {
    @TableId(value = "id", type = IdType.ASSIGN_ID)
    private Long id;
    // ...
}

// Role.java
@Data
@TableName("ams_role")
public class Role implements Serializable {
    @TableId(value = "id", type = IdType.ASSIGN_ID)
    private Long id;
    // ...
}
```

**修改文件列表**：
- `mall-pojo/admin/model/Admin.java`
- `mall-pojo/admin/model/Role.java`
- `mall-pojo/admin/model/Permission.java`
- `mall-pojo/admin/model/RolePermission.java`
- `mall-pojo/admin/model/AdminRole.java`
- `mall-pojo/admin/model/AdminLoginLog.java`

## 三、MyBatis Plus Generator API 变更 (3.x → 4.x)

### 3.1 naming() 和 columnNaming() 方法位置变更

**问题描述**：
```
Cannot resolve method 'naming' in 'Builder'
```

**根本原因**：
MyBatis Plus Generator 4.x 中，`naming()` 和 `columnNaming()` 方法必须在 `entityBuilder()` 内部调用。

**解决方案**：
```java
// 修复前（3.x 写法）
builder.naming(NamingStrategy.underline_to_camel)
       .columnNaming(NamingStrategy.underline_to_camel)
       .addInclude(...)
       .entityBuilder()
           ...

// 修复后（4.x 写法）
builder.addInclude(...)
       .addTablePrefix(...)
       .entityBuilder()
           .naming(NamingStrategy.underline_to_camel)
           .columnNaming(NamingStrategy.underline_to_camel)
           ...
```

---

### 3.2 formatName() 方法已移除

**问题描述**：
```
Cannot resolve method 'formatName' in 'Builder'
```

**根本原因**：
MyBatis Plus Generator 4.x 移除了 `formatName()` 方法。包路径应该通过 `packageConfig` 中的配置来设置。

**解决方案**：
移除 `formatName()` 调用，删除未使用的导入：
```java
// 移除
.formatName("%s", i -> "model." + i)
.formatName("%s", i -> "web.controller." + i)

// 同时清理未使用的导入
import com.baomidou.mybatisplus.core.exceptions.MybatisPlusException;
import com.baomidou.mybatisplus.core.toolkit.StringPool;
import com.baomidou.mybatisplus.core.toolkit.StringUtils;
import java.sql.Types;
import java.util.ArrayList;
import java.util.List;
```

## 四、类导入路径错误

### 4.1 CsmallAuthenticationInfo 导入路径错误

**问题描述**：
```
Cannot resolve symbol 'CsmallAuthenticationInfo'
```

**根本原因**：
`CsmallAuthenticationInfo` 类的正确包路径是 `com.cooxiao.mall.common.domain`，但代码中错误地使用了 `com.cooxiao.mall.common.pojo.domain`。

**解决方案**：
将所有文件中的导入路径从 `com.cooxiao.mall.common.pojo.domain.CsmallAuthenticationInfo` 修改为 `com.cooxiao.mall.common.domain.CsmallAuthenticationInfo`。

**修改文件列表**：
- `mall-ams`: SSOFilter.java, SecurityContextUtils.java
- `mall-front`: SSOFilter.java
- `mall-order`: SSOFilter.java, OmsOrderServiceImpl.java, OmsCartServiceImpl.java
- `mall-seckill`: SSOFilter.java, SeckillServiceImpl.java

## 五、其他小问题

### 5.1 JJWT 版本号硬编码

**问题描述**：
JJWT 依赖在子模块中硬编码了 `0.9.0` 版本号，与父 POM 管理的 `${jjwt.version}` 不一致。

**解决方案**：
移除硬编码的版本号，让其继承父 POM 的版本管理。

```xml
<!-- 修复前 -->
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt</artifactId>
    <version>0.9.0</version>
</dependency>

<!-- 修复后 -->
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt</artifactId>
</dependency>
```
