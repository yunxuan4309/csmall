# 技术栈二次更迭遇到的问题

> 时间：2026-05-09
> 内容：弃用 mall-leaf 模块，迁移至 MyBatis-Plus IdWorker 分布式 ID 生成

---

## 一、问题背景

mall-leaf 模块基于 JDK 8 + Spring Boot 2.5.4 + Spring 5.x，与项目其他模块（JDK 21 + Spring Boot 3.2.5）存在代差，无法统一构建。根 pom 中已有注释：`聚合执行命令会因为leaf存在失败`。

经评估，美团 Leaf 开源项目自 2020 年起已停止维护，上游无适配 JDK 21 的计划。且项目已使用 MyBatis-Plus `IdType.ASSIGN_ID` 作为主要 ID 策略，Leaf 服务并非必需。

---

## 二、发现的问题

### 1. Leaf 调用方式不一致

| 模块 | 方法名 | 端口 | 是否实际调用 |
|------|--------|------|------------|
| mall-order | `getDistributeId()` | 9090 | 是（2处） |
| mall-product | `getDistributeId()` | 9090 | 是（2处） |
| mall-ums | `getDistributeId()` | 9090 | 是（1处） |
| mall-ams | `generatId()`（拼写错误） | **8080**（错误） | 是（1处） |
| mall-front | `generatId()` | **8080**（错误） | 否（死代码） |
| mall-search | `getDistributeId()` | 9090 | 否（死代码） |

- mall-ams 和 mall-front 的端口配置错误（8080 vs 9090），运行时必然失败
- 方法名不统一：`getDistributeId` vs `generatId`

### 2. 自定义 XML 插入绕过 MyBatis-Plus ID 生成

6 个实体类均声明了 `@TableId(type = IdType.ASSIGN_ID)`，但自定义 XML Mapper 的 insert 方法显式包含 `#{id}`，导致 ASSIGN_ID 注解实际无效。ID 由 Leaf HTTP 调用生成后手动 `setId()`。

---

## 三、解决方案

### 替换方案：`IdWorker.getId()`

`IdWorker` 是 MyBatis-Plus 内置的雪花 ID 生成工具，专为自定义 XML 插入场景设计，与 `IdType.ASSIGN_ID` 使用相同的算法，无需外部服务。

| 原调用 | 替换为 |
|--------|--------|
| `IdGeneratorUtils.getDistributeId("order")` | `IdWorker.getId()` |
| `IdGeneratorUtils.getDistributeId("order_item")` | `IdWorker.getId()` |
| `IdGeneratorUtils.getDistributeId(Key.SPU)` | `IdWorker.getId()` |
| `IdGeneratorUtils.getDistributeId(Key.SKU)` | `IdWorker.getId()` |
| `IdGeneratorUtils.getDistributeId("user")` | `IdWorker.getId()` |
| `IdGeneratorUtils.generatId("admin")` | `IdWorker.getId()` |

### 具体改动

1. **4 个 Service 文件**：替换 `IdGeneratorUtils` 调用为 `IdWorker.getId()`，import 从 `xxx.utils.IdGeneratorUtils` 改为 `com.baomidou.mybatisplus.core.toolkit.IdWorker`
2. **删除 6 个 `IdGeneratorUtils.java`**：mall-order、mall-product、mall-ums、mall-ams、mall-front、mall-search
3. **根 pom.xml**：注释掉 `<module>mall-leaf</module>` 和 mall-leaf 依赖管理
4. **新增 `mall-leaf/DEPRECATED.md`**：弃用说明文档

---

## 四、方案优势与劣势

### 优势
- 消除外部依赖，不再需要单独启动 Leaf 服务
- 统一技术栈，所有模块基于 JDK 21
- 进程内生成 ID，比 HTTP 调用更快
- 修复了 mall-ams 端口配置错误（8080 vs 9090）
- 部署时少一个服务，降低运维成本

### 劣势
- ID 不再趋势递增（雪花算法是趋势递增但不连续）
- Long 精度问题：已通过 `mall-common` 的 `JacksonConfiguration` 全局配置 Long→String 解决

---

## 五、mall-leaf 模块现状

- 源码保留在仓库中，不删除
- 不再参与 Maven 聚合构建
- 详细弃用说明见 `mall-leaf/DEPRECATED.md`
